## What shipped

**Guards (`lib/access.ts`).** Pure `employeeAreaRedirect`,
`portalAreaRedirect`, and `canViewAccount` — the boundary as functions, so it
is pinned by tests rather than by hidden links. The `(employee)` layout
redirects a CLIENT to the portal and a parked USER to login before rendering
any child route; the `(client)` layout sends non-client/non-admin to the hub;
the portal reads call `canViewAccount`.

**Portal reads (`lib/queries/portal.ts`).** `portalData(accountId, viewerId)`
returns only the client-safe projection: `ClientJobView` carries gross value
and a plain-language status label — never `firmMargin`, `workerPool`, bids, or
the raw stage — and every read is scoped to the one account passed in
(resolved from the session, never a URL id). `sharedAssetsForAccount`
(from M6) is the leak-proof asset scope. The contact is the won-deal owner,
else the managing director.

**Actions.** `profile.ts` (`updateProfile`, own row); `onboarding.ts`
(`dismissPortalStart`, `confirmPortalDetails`, `markBriefReviewed` — CLIENT,
each a one-way null-guarded stamp, now plus `finalizeJoblessSetup`);
`portal.ts` (`publishGuide`, `upsertInfoBarMessage`, `toggleInfoBarMessage`,
`reorderInfoBar` — MODERATOR/ADMIN); `admin.ts` (`updateUserRole`,
`updateUserDepartment` with confirmation copy, `createEmployee` landing
onboarding-pending — ADMIN).

**UI.** Both portal compositions on live data, switched by `Account.kind` on
the server. Business: figures row (gross only), contact card, engagements
table in client language, account thread with the review-call prefill, and
the start-here panel with a one-way Dismiss. Individual: the three-step setup
strip (step 3 reviews the most recent engagement or auto-satisfies when there
is none), one engagement card with the brief and delivered assets inline, the
prominent thread, no figures or contact card. Shared foundation: rotating
info bar, the guide in Newsreader, masthead + wing hairline. Settings, tiered
and live: profile for everyone (email/role read-only), guide editor with an
autosaved local draft and live Newsreader preview plus info-bar CRUD for
MODERATOR/ADMIN, and the user matrix + Add employee for ADMIN.

## Decisions and boundaries

1. **The portal has no account id in the URL.** Scoping is by the session
   `accountId` (or the admin-preview constant), so guarantee 3 holds
   structurally — there is no id to tamper with.
2. **OPEN jobs are not the client's business.** The portal job query excludes
   OPEN, so a posting that isn't yet staffed doesn't surface; engagements
   appear once scheduled.
3. **The job brief (description) is shown to individual clients** per §7.13 —
   authored client-side by admins who know it is client-visible.
4. **The employee first-week welcome page and `completeOnboarding` are M8**,
   not M7. `createEmployee` correctly lands the hire with `onboardingCompletedAt`
   null — the M7 done-when.
5. **`/settings` self-guards** (it sits outside the route groups): no session →
   login, parked USER → login; the editor/matrix tiers render only for their
   roles and every action re-checks the role server-side.

## Adversarial pass (leakage-focused)

Four bounded auditors — the three isolation guarantees; the M7 actions and
conditional writes; the onboarding edge and the step-3 ruling; PRD/design —
each finding adversarially verified by an independent refuter. 15 agents, 11
raw findings, 4 confirmed, 7 refuted. **Every leakage-auditor finding was
refuted**: no client can reach an employee route (the layout redirect fires
before any child renders), reach internal figures (the projection selects
none and the employee job route redirects clients), or reach another account
(scoping is server-side, and `canViewChannel` already refuses cross-account
threads). The four confirmed, all fixed before commit:

- **MEDIUM — info-bar create assigned `order` non-atomically.** The append
  read the current max and wrote max+1 outside a transaction, so two
  concurrent creates could collide on one order. Now the max-read and create
  run under Serializable with a bounded retry (reorder too); the other three
  info-bar writes were already guarded.
- **MEDIUM — the individual setup strip could re-expand after completion.**
  Completion was derived live, so a job-less client who collapsed the strip
  (step 3 auto-satisfied, `briefReviewedAt` never stamped) would see it
  re-open when the firm later staffed their first engagement. `finalizeJoblessSetup`
  now durably stamps `briefReviewedAt` the moment a job-less individual
  finishes the two interactive steps, so finishing is final — pinned by a
  regression test.
- **LOW — `/settings` rendered a dead form for a parked USER.** Added the
  USER→login guard the rest of the app already enforces.
- **LOW — matrix selects desynced from the DB on a rejected change.** The
  uncontrolled `<select>` kept the rejected value when an action failed (e.g.
  an admin demoting their own last admin); `router.refresh()` now runs in both
  branches, repainting the control to the database value.

Refuted, for the record: the guarantee-1 "needs an integration test" (the
spec's stated bar is the M4-style pure helper test, which is pinned); admin
setting a department on a CLIENT (schema-permitted, no leak — `canViewChannel`
ignores a client's departmentId); a "torn portal view" from `portalData`'s
read assembly (read-only, single-account-scoped, no corruption); `/settings`
for a USER as a leak (no leak — own data only, action refused; the consistency
gap is the confirmed low above); employee onboarding "unimplemented" (M8); the
float `TOTAL INVESTED` sum (`formatMoney` rounds to the cent — divergence
provably impossible for two-decimal inputs); and `confirmPortalDetails`'s
read-then-write (date-only display, no observable effect).
