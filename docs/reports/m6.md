# M6 report — Vault, graph, leaderboards, forum

Scope: PRD 7.4 (forum), 7.5 (vault), 7.6 (knowledge graph), 7.7
(leaderboards), the §9 action rules, and the M6 acceptance list in §13.
Built against the spec as amended by the ratifications that arrived with
the M5 acknowledgment, folded into PRD.md first (commit 7a1d077) — §7.11
(convertWonDeal validates before it writes), §7.12 (claim-copy ruling and
the inbound-message-at-receipt timestamp), §7.7 (contributor exclusion of
Shadow output and system users), §7.6 (system users are never person
nodes). docs/ui-contract.md was untouched: no amendment reaches §5.

## What shipped

**Domain (`lib/`).** `lib/leaderboards.ts` — three boards by indexed
`groupBy` queries, no stored scores: jobs executed (all-time / 90-day by
`completedAt`), gross value worked (full gross credited per worker on team
jobs), knowledge contributors (trailing 30 days; messages excluding any
with `approvedById` set, Shadow drafts, and system users, +3×forum, +2×
lessons — the pre-M6 ruling). `lib/graph/build.ts` — a pure `deriveGraph`
holding every §7.6 rule (departments, employees, all jobs, accounts-with-
jobs, commons-or-delivered assets; edges person—department / person—job /
account—job / asset—job, no pairwise person—person; the 300-node cap that
collapses assets first and reports the count) with `buildGraph` fetching
and serializing on top. `lib/progression.ts` gained `awardForumXp` — the
daily-cap-aware award (5 per post/reply, max 25/day), an aggregate guard
run under the contributor's row lock.

**Actions.** `app/actions/vault.ts` — `addVaultAsset` (metadata + external
URL, the form states storage is V3) and `toggleAssetSharing` (conditional
write on the read value, so a racing flip resolves cleanly).
`app/actions/forum.ts` — `createForumPost` and `replyToPost`, both posting
with capped XP inside a Serializable transaction that locks the User row
before summing the day's forum events, then surfaces any tier crossing for
the quiet toast. Replies are forced one level deep (a reply to a reply
attaches to the top-level post).

**UI.** Vault: the live table (title, type, linked job, uploader, date,
size, a per-row Commons toggle), type / department / commons filters in the
query string, and a full-width Add asset band. Graph: the page derives on
the server and hands nodes/edges to a `"use client"` `GraphCanvas` running
a `d3-force` layout in plain React SVG — pan by drag, wheel-zoom 0.5–2.5×,
hover highlights the 1-hop neighborhood (the rest fade to 25%), click opens
the inspector with the node's facts and a link into the app; the simulation
settles, auto-fits the view, and stops; `prefers-reduced-motion` ticks it
to rest before the first paint. Leaderboards: three ranked tables with tier
badges on the acting members, the all-time / 90-day toggle, the gross-value
tooltip, rows linking to profiles. Forum: the live reverse-chrono feed with
a full-width New post composer and per-post one-level reply composers.

## Decisions and boundaries

1. **Graph node coverage.** §7.6 lists "jobs" without qualifying status, so
   all jobs are nodes — COMPLETED renders `ok`, every other status renders
   `warn` (in-flight). Accounts appear only when a job commissions them
   (prospects with no jobs stay off). Assets appear when in the Commons or
   delivered (their job COMPLETED). System users and client/parked users are
   excluded from person nodes — person nodes are employees (§7.6 line 616,
   the role filter `EMPLOYEE | MODERATOR | ADMIN`).
2. **Account node size/colour are the PRD's silent fill-ins.** §7.6 gives
   sizes for department/job/person/asset and a colour vocabulary but does
   not size or colour the client-account node. I used 24px (a commissioning
   hub between job and department) and the `info` token. Flagged here as a
   choice, not a spec quote.
3. **Account-status fact shows the raw enum** ("ACTIVE") in the inspector,
   matching the existing CRM account pages (which render the bare uppercase
   token by design); job status humanizes to sentence case as elsewhere.
4. **The portal shared-assets panel stays on the M0 mock until M7.** The
   live, leak-proof `sharedAssetsForAccount` query is written and tested-by-
   construction (account-scoped, shared-or-REVIEW/COMPLETED only), ready for
   M7 to consume; rebuilding the portal page is the named M7 deliverable, so
   it is out of M6 scope.
5. **Forum daily cap is per server-local day.** The PRD names no firm
   timezone; the cap resets once per local day and never mid-day, so the
   only stated invariant (25/day) holds. A pinned firm timezone would be a
   V3 hardening item.

## Adversarial pass

Four bounded auditors (auth/leakage + the exclusion ruling; concurrency /
the forum XP cap / conditional writes; the graph derivation and client; PRD
conformance + design rules) ran as subagents over fixed file scopes, each
finding adversarially verified by an independent refuter — 14 agents, 10
raw findings, 2 confirmed, 8 refuted. Both confirmed, plus one genuine
resilience gap the refuter surfaced in passing, were fixed before commit:

- **MEDIUM — composer forms cramped into the header.** The vault Add asset
  and forum New post panels were passed into `PageHeader`'s `actions` slot,
  so their open state rendered as a shrink-to-fit form in the header's right
  cell instead of a full-width band below it (the accepted M2 NewDealPanel /
  NewPostingPanel pattern). Moved both out of `actions` to sibling bands;
  their closed state is now a band with the button. Verified live: the open
  composer spans the full main width below the header.
- **LOW — forum tier-up toast could not fire.** `postWithCappedXp` discarded
  `awardForumXp`'s tier result, so a forum post crossing a tier boundary
  produced no toast where §7.2 calls for one (the one XP source that
  couldn't). Threaded the tier-up back through both actions; the composers
  now push the gold "Tier N — name" toast like the academy and CRM paths,
  and a tier change revalidates the layout.
- **(surfaced during refutation) — the forum retry was dead for its own
  lock.** The `SELECT … FOR UPDATE` aggregate lock aborts under Serializable
  as Prisma `P2010` (raw-query) carrying PG `40001`, not `P2034`, so the
  single-`P2034` retry never engaged and a near-simultaneous same-user
  double-submit could drop a post with a generic error. The forum guard now
  retries on either serialization signal, bounded to three attempts.

Refuted, for the record: the portal-on-mock "staleness" (the M7 boundary,
spec-prescribed); a missing `/client-portal` revalidation on `addVaultAsset`
(the portal reads no DB vault rows in M6); the contributor board ranking
CLIENT users (matches §7.7 exactly — only Shadow output and system users are
excluded); the "single retry insufficient" thesis (rested on a P2034/P2010
misread — but pointed at the real gap above); the forum-cap timezone
(unspecified by the PRD, invariant holds per process); the role-vs-isSystem
person filter (person nodes are employees, and an isSystem-only filter would
wrongly map all six seeded CLIENT contacts onto the delivery map); the
300-cap "non-asset overflow" (~61 max non-asset nodes in a sub-30-person
firm — unreachable); and the raw account-status enum (the established CRM
convention).
