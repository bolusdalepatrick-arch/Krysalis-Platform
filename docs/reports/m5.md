# M5 report — CRM & the gate loop

Scope: PRD 7.11 (CRM), 7.12 (Bounty Board and webhooks), the section 9
catalog rows they own, and the M5 acceptance list in section 13. Built
against the spec as amended by the pre-M5 rulings, which were folded into
PRD.md first (commit de4517a) — section 7.3 (approval re-timestamp,
archived channels refuse drafts, department transparency, Shadow source

## What shipped

**Domain (`lib/`).** `lib/hmac.ts` — hex HMAC-SHA256 sign plus timing-safe
verify over the raw body, `node:crypto` only. `lib/crm.ts` — the stage
rules (`validateStageMove`, `stageChangeBody`), the case-insensitive
account find-or-create, and the two transaction bodies actions and tests
share, following the `lib/academy.ts` pattern: `applyClaim` (conditional
UNCLAIMED → CLAIMED, then account/contact/deal/follow-up in the same
transaction) and `applyConversion` (validate everything, then write — a
failed conversion commits nothing). `lib/webhooks.ts` — the outbound claim
notification with the gate's mock semantics, a 10s timeout, and the "409"
mock-failure convention. Zod schemas for every new action plus the inbound
payload in `lib/validators.ts`.

**Actions (`app/actions/crm.ts`).** `createDeal` (existing account or new
account + contact, Serializable), `updateDeal` (owner/ADMIN, open deals
only, ownership inside the conditional WHERE), `setDealStage` (stage-guarded
conditional write, automatic STAGE_CHANGE, WON stamps `wonAt` and awards
DEAL_WON through `lib/progression.ts` to the owner re-read inside the
transaction, LOST requires the plain-sentence reason), `logDealActivity`,
`claimBookingCard` (Serializable claim transaction; post-commit signed
notification that can never convert the committed claim into a reported
failure), `resendClaimWebhook` (ADMIN), `archiveBookingCard` (ADMIN,
conditional UNCLAIMED → ARCHIVED on expired cards only), `convertWonDeal`
(ADMIN, Serializable, re-runnable find-or-skip).

**The one route handler.** `app/api/hooks/booking/route.ts`: 401 on bad
signature before anything else runs, 422 on JSON or zod failure, idempotent
upsert on `externalRef`, inline card message posted under the Gate user to
`#new-business` (stamped now — the poll cursor only looks forward; the
card row keeps the visitor's `submittedAt`), revalidation, `{ ok: true }`.
`npm run simulate:booking` signs the verbatim section 7.12 sample and POSTs
it locally; fixed bookingId makes it idempotent across runs. `.env.example`
now ships the marked `dev-not-a-secret` placeholder, an empty
`N8N_CLAIM_WEBHOOK`, and `MOCK_WEBHOOKS` unset, per the ruling.

**UI.** Pipeline on live data: dense table default — all eight columns
sortable via header links, stage and owner filters — board toggle with
single-line rows; "New deal" panel (existing or new account). Deal detail:
stage rail, stage controls with the value guard sentence, facts column with
inline editor (value, expected close, ADMIN owner reassign), activity log
with composer (NOTE/CALL/EMAIL/MEETING), contact card, linked booking and
engagement, and the ADMIN conversion panel on WON deals (kind select,
portal user on by default with exclude, the job draft confirmed never
silent, gross locked to the deal's value). Account list and detail live,
portal users shown for ACTIVE accounts. Bounty board: unclaimed cards
first (admin Archive on expired ones), an admin "Outbound failures"
section with "Resend to n8n", then claim history as a table. Booking cards
render inline in `#new-business` with one live Claim button — pending
state, no optimism; the loser's "Already claimed by {name}." arrives as a
toast that outlives the refresh which flips the card to its claimed state
("Claimed by June Park · Jun 12, 14:08", muted mono, badge-free). The
Today view's PIPELINE and UNCLAIMED BOUNTIES blocks moved off mock onto
live queries; `lib/queries/crm.ts` carries all CRM reads.

## Decisions and PRD conflicts to surface

1. **Claim follow-up copy — PRD-vs-code conflict, ruling requested.** PRD
   7.12 specifies the in-transaction follow-up as "Claimed by June Park —
   deal opened." The implementation posts "Claimed — deal opened under
   {account}." under the claimer, matching the register the committed M1
   seed already established (`prisma/seed-data.ts:737`). Runtime and seed
   are consistent with each other and both deviate from the PRD sentence.
   Either bless the seed shape in 7.12 or both get aligned to the spec.
2. **Decided deals are terminal.** The PRD specifies movement between the
   four working stages and the WON/LOST rules but is silent on reopening.
   `validateStageMove` rules WON and LOST terminal ("A decided deal doesn't
   move stages. Open a new deal for repeat business."), consistent with the
   repeat-business ruling. `updateDeal` likewise refuses decided deals —
   a converted deal's value must keep equaling its job's gross (the seed
   integrity invariant).
3. **Manual deals open at INBOUND** (schema default; 7.11 names no initial
   stage for manual entry).
4. **Inbound card messages are stamped at receipt, not `submittedAt`.**
   A backdated message would sit behind the 5s poll cursor and never reach
   open windows — the same reasoning as the ratified approval re-timestamp.
   The card row keeps `submittedAt`; only the chat row is stamped now.
5. **Archived cards leave the bounty board entirely** (7.12 lists only
   unclaimed and recent claims). Their inline channel messages remain, in
   the muted "Archived — the slot passed unclaimed" state.
6. **DEAL_WON goes to the deal's owner**, re-read inside the transaction;
   the tier-up toast fires only when the actor is the owner.
7. **`Account.name` stays schema-unique (case-sensitive).** True
   case-insensitive uniqueness would need a functional index and a section
   8 amendment; Serializable isolation on every find-or-create path
   (claim, createDeal, conversion) closes the duplicate-account race
   without touching the "complete and final" schema.

## Adversarial pass

Four bounded auditors (concurrency/transactions, authorization/leakage,
PRD conformance, design rules) ran as subagents over fixed file scopes,
each finding then adversarially verified by an independent agent told to
refute it — 21 agents, 17 raw findings, 12 confirmed, 5 refuted. The 12
confirmed de-duplicate to 8 distinct issues (two were found independently
by two auditors). All were fixed before commit, except item 1 above, which
is a conflict to rule on, not code to change silently:

- **HIGH — failed conversion committed partial state.** `applyConversion`
  validated the draft job and the contact collision after the ACTIVE flip,
  thread create, and portal-user create; a `converted: false` outcome
  still committed them. Restructured: every failure path now precedes the
  first write, pinned by a regression test.
- **HIGH — backdated inbound message defeated the poll.** Fixed per
  decision 4 (found by the auditor before it was found by demo: the dev
  machine's timezone had masked it).
- **HIGH — `createDeal` ran its find-or-create at Read Committed**, racing
  duplicate accounts and duplicate primary contacts against the
  Serializable claim path. Now Serializable; `guarded` already retries
  P2034 once.
- **MEDIUM — post-commit failures reported a committed claim as failed.**
  The notification block after the claim commit is now best-effort: a
  throw there logs and the claimer still gets the truth. The residual gap
  the auditor noted — a process crash between commit and the error write
  leaves no resend affordance — is inherent to 7.12's post-commit design;
  a pending-delivery marker inside the claim transaction is the suggested
  future amendment if it matters operationally.
- **MEDIUM — no timeout on the outbound fetch.** `AbortSignal.timeout(10s)`.
- **MEDIUM — claimed-state line dropped the claim time** the PRD shows
  ("Jun 12, 14:08"); it rendered the year instead. Now `formatChatTime`.
- **LOW — ownership checks were read-then-write** in `updateDeal` and
  `setDealStage`; ownership now sits inside the conditional WHERE for
  non-admins.
- **LOW — `simulate:booking` used `process.loadEnvFile`** (Node 20.12+)
  against the repo's 20.9 floor; replaced with a hand parser.

Refuted, for the record: a claimed missing layout revalidation (all routes
are fully dynamic — nothing cached to go stale, and the actor's window
refreshes), a claimed P2002 path on concurrent conversions (empirically
P2034, which is retried), a missing per-page auth check (the route-group
layout is the spec'd guard, per changes-log item 21), the claim-history
table "contradicting" 7.12 (the section 6 file map specifies "unclaimed
cards + claim history", and hard rule 5.1.9 mandates the table), and a
~200-line cap breach at 219 lines (the tilde, and the committed 227-line
baseline).
